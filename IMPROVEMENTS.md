# LocalCoder 改善提案

> 文書の役割: 今後の改善候補と優先順位を管理するロードマップ。ここに記載された項目は、明示的に実装済みと記録されたものを除いて計画として扱う。現在の実装状況は README.md と CHANGELOG.md を参照する。

LocalCoder を、限界のあるローカルLLMでも長時間・実作業で安定して使えるコーディングエージェントへ発展させるための改善案。

不可逆な操作を抑制する安全設計については [REVERSIBLE_OPERATIONS.md](REVERSIBLE_OPERATIONS.md) を、作業方針の自動再評価（停滞・目的逸脱の検知とCONTINUE/ADJUST/CHANGE/STOP判定）については [METACOGNITIVE_REPLANNING.md](METACOGNITIVE_REPLANNING.md) を参照する。本書では、それ以外の信頼性、観測性、テスト、性能、操作性、保守性、配布運用に関する改善を扱う。

外部の専門知識MCPサーバー（例: PicoCalc開発支援エキスパートMCP、設計文書は開発者のGoogle Drive上）をLocalCoderへ接続する構想があり、その文書はLocalCoder側に特定の前提（テスト基盤・状態管理・ツール実行の分離）を明示的に要求している。この前提と本書の関係は §13 にまとめる。

## 1. 改善方針

LocalCoder の強みは、高性能なモデルを前提とせず、モデルが止まる、忘れる、値を誤る、同じ操作を繰り返すという実際の失敗をサーバー側の仕組みで補正している点にある。

今後も次の原則を維持する。

- モデルの能力向上だけに依存しない
- 重要な状態はLLMの記憶ではなく機械的に保持する
- 障害は隠さず記録し、再現可能にする
- 自律性を保ち、不要な確認を増やさない
- 実測した失敗から対策を決める
- 基本機能はローカルで完結させる

## 2. 最優先改善

### 2.1 履歴圧縮と復旧処理の自動テスト

履歴圧縮は長時間作業を成立させる中核機能である。固定入力による回帰テストを追加し、変更による再発を防ぐ。

最低限、次をテストする。

- 初回圧縮
- 2回目以降の増分圧縮
- 既存要約の保持
- 変更ファイル一覧の機械的保持と重複除去
- 要約失敗時の既存情報保持
- 強制圧縮
- 90%閾値の境界
- 古いツール結果の切り詰めと重複除去
- 画像付きメッセージのトークン見積もり
- 空応答後の強制圧縮と再試行
- 特殊トークン混入時のツール名正規化
- 同一ツール呼び出しの停止判定

```text
tests/
├── test_token_estimation.py
├── test_history_compaction.py
├── test_work_state.py
├── test_tool_normalization.py
└── fixtures/
    ├── long_session.json
    ├── repeated_compaction.json
    └── leaked_special_token.json
```

Ollamaを必要としない純粋関数からテストし、LLM要約部分は固定応答へ差し替える。

### 2.2 実障害セッションの回帰フィクスチャ化

実際に問題が起きた履歴を、個人情報とパスを匿名化してテスト入力として保存する。

- 99%のコンテキスト使用率で空応答になったセッション
- 圧縮の繰り返しでファイルパスが失われたセッション
- 同じコマンドを繰り返して停止したセッション
- 特殊トークンがツール名へ混入したセッション
- 古いサーバープロセスを使い続けていた事例

各障害について、修正前には失敗し、修正後には成功する再生テストを置く。

### 2.3 診断情報の保存

各ターンについて次を保存する。

- 開始・終了時刻
- 終了状態
- モデル名
- 推定入力トークン数
- 圧縮前後の推定トークン数
- 圧縮回数
- Ollama再試行回数
- 空応答回数
- ツール呼び出し回数
- コマンド合計実行時間
- 最終終了理由

履歴画面から診断情報を確認できるようにし、障害解析を会話ログの目視だけに依存させない。

## 3. 状態管理

### 3.1 現在のゴールを独立保持

会話要約とは別に、現在の作業目標を固定フィールドで保持する。

```json
{
  "goal": "既存APIとの互換性を保ったまま履歴圧縮を修正する",
  "done_conditions": [
    "既存テストが通る",
    "再現セッションで空応答が起きない"
  ]
}
```

ゴールを要約本文へ埋め込まず、圧縮を繰り返しても必ず残る構造にする。

### 3.2 ユーザー指示のピン留め

「変更しないで」「この値を使う」「外部送信しない」などの継続指示を、LLM要約の外側に逐語保存する。

- 原文を保持する
- 作成時刻と発言位置を記録する
- 上書きまたは解除された場合だけ無効化する
- 毎回のモデル呼び出しでsystem prompt直後に挿入する

### 3.3 検証条件の機械的管理

モデルが「完了した」と述べただけで完了扱いにせず、タスクごとの検証条件を保持する。

```text
変更したファイルがある
    → ビルドまたはテスト結果が必要

Web UIを変更した
    → HTML/JSの構文確認が必要

設定を変更した
    → 読み戻して値を確認
```

検証未実施なら、モデルへ未完了状態として明示する。

## 4. ツール実行の信頼性

### 4.1 構造化されたツール結果

現在の文字列結果に加えて、内部では構造化データを保持する。

```json
{
  "ok": false,
  "exit_code": 1,
  "stdout": "...",
  "stderr": "...",
  "duration_ms": 1834,
  "timed_out": false,
  "cancelled": false
}
```

モデルには読みやすい文字列を渡し、サーバー側では機械判定に構造化データを使う。

### 4.2 出力の全文保存と表示用切り詰めの分離

モデルへ渡す出力は切り詰めても、完全なstdout/stderrは診断ログへ保存する。

```text
history/<sid>/tool-results/<call-id>.json
```

これにより、12KBを超えた出力の後半や途中に重要なエラーがあっても、後から確認できる。

### 4.3 長時間コマンドの進捗通知

ビルドやテストが長時間無出力でも停止と誤認しないよう、一定間隔で次をSSE通知する。

- 実行経過時間
- 最終出力時刻
- 直近の出力
- タイムアウトまでの残り

可能ならstdout/stderrを逐次配信する。

### 4.4 タイムアウトのツール別設定

全コマンド一律180秒ではなく、用途別に上限を設定可能にする。

```text
search: 60秒
build: 600秒
test: 600秒
server: 明示停止まで
```

`run_command`に任意の無制限値を渡させず、設定済みプロファイルから選択する。

### 4.5 バックグラウンドプロセス管理

開発サーバー等を起動する専用ツールを追加する。

- `start_process`
- `process_status`
- `read_process_output`
- `stop_process`

PIDだけでなくプロセスグループ、起動コマンド、作業フォルダ、開始時刻を管理する。

## 5. モデル適応

### 5.1 モデル能力プロファイル

vision対応だけでなく、モデルごとに次を記録する。

- tool calling対応
- 推奨コンテキスト長
- JSON引数の安定性
- 空応答発生率
- 平均ツール反復回数
- 推奨temperature
- 画像対応

実測値をローカルに蓄積し、モデル選択UIへ表示する。

### 5.2 モデル別プロンプト

すべてのモデルへ同一の長いsystem promptを送るのではなく、失敗傾向に応じて補助指示を切り替える。

- ツール引数を欠落しやすいモデル
- 終了を急ぐモデル
- 同じ操作を繰り返すモデル
- 長文脈で空応答になりやすいモデル

### 5.3 自動フォールバック

選択モデルが連続して空応答、ツール形式エラー、HTTP失敗を起こした場合、事前指定した代替モデルへ切り替えて継続する。

切り替え時は、作業状態、現在のゴール、直近メッセージを引き継ぐ。無断切替を望まない場合に備え、設定で無効化可能にする。

## 6. コンテキストと性能

### 6.1 正確なトークン計測への移行

現在の文字数ベース概算を残しつつ、Ollamaまたはモデル側から利用可能なトークン情報が得られる場合は実測値を優先する。

概算と実測の差を記録し、モデル別の補正係数を学習する。

### 6.2 読み取り結果の参照化

巨大ファイルや長いコマンド出力を毎回会話へ埋め込まず、内部参照として保存する。

```text
[artifact:read-42]
path=src/large_file.cpp
lines=1-1800
sha256=...
```

モデルへ必要部分だけ再提示し、同じ内容の再送を減らす。

### 6.3 差分中心の再読

一度読んだファイルについて、変更後は全文ではなく差分と周辺行を優先して渡す。ファイルのSHA-256を保持し、内容が変わっていない場合は再読結果を再利用する。

### 6.4 キャッシュ無効化の可視化

履歴圧縮やsystem prompt変更でOllamaのプロンプトキャッシュが無効になった可能性を診断ログへ記録する。応答遅延の原因をGPU性能だけに誤認しないようにする。

## 7. UI・操作性

### 7.1 ターン単位の作業サマリー

各ターン終了時に次を表示する。

- 変更ファイル
- 実行コマンド
- テスト結果
- 圧縮回数
- 所要時間
- 終了理由
- 未完了事項

### 7.2 ライブ作業状態

モデルが現在何をしているかを簡潔に表示する。

```text
調査中
3ファイル確認済み
ビルド1回失敗
現在: リンクエラーの原因を検索中
```

モデルの思考内容ではなく、ツール実行から機械的に導出した状態だけを表示する。

### 7.3 セッション内検索

会話、ツール呼び出し、変更ファイル、コマンド出力を横断検索できるようにする。

### 7.4 モデル切替時の明示

途中でモデルを変更した場合、履歴へ境界マーカーを残す。

```text
モデル変更: gpt-oss:20b → qwen3-vl:8b
理由: 画像確認
```

### 7.5 設定のエクスポート・インポート

次をJSONで保存・復元できるようにする。

- Ollama URL
- 既定モデル
- 作業フォルダ
- allowed roots
- コンテキスト設定
- 外部送信ポリシー
- モデル別設定

## 8. 保守性

### 8.1 `server.py`の段階的分割

単一ファイルの利点を残しつつ、変更頻度と責務が明確になった部分から分割する。

```text
localcoder/
├── server.py
├── agent.py
├── tools.py
├── history.py
├── compaction.py
├── ollama.py
├── sessions.py
└── security.py
```

一度に全面再構成せず、テストを追加した後に純粋関数群から移動する。

### 8.2 設定値の集約

環境変数と定数を`Config`へ集約し、起動時に検証する。

- 数値範囲
- パスの存在
- allowed rootsの正規化
- Ollama URL
- タイムアウト
- コンテキスト予算

起動時に有効設定をログへ出す。

### 8.3 セッション形式のバージョン管理

履歴JSONへ`schema_version`を追加する。

```json
{
  "schema_version": 2,
  "sid": "..."
}
```

形式変更時は読み込み時に移行し、古い履歴を壊さない。

### 8.4 ログと履歴の分離

ユーザー向け会話履歴と、内部診断ログを別ファイルにする。会話履歴を読みやすく保ちつつ、障害解析に必要な情報を失わない。

## 9. 配布と運用

### 9.1 バージョンタグとGitHub Release

機能単位でバージョンを付け、CHANGELOGと対応させる。

- タグ
- 主要変更
- 既知の問題
- 更新手順
- 対応Ollamaバージョン
- 動作確認モデル

### 9.2 起動時セルフチェック

起動時に次を確認し、GUIへ結果を表示する。

- Ollamaへ接続可能か
- 推奨モデルが存在するか
- `pdftotext`が存在するか
- allowed rootsが有効か
- historyへ書き込み可能か
- ポートが利用可能か
- Windows/WSL相互運用が利用可能か

### 9.3 診断情報の一括出力

問題報告用に、秘密情報を除去した診断パッケージを生成する。

- LocalCoderバージョン
- OS/WSL情報
- Pythonバージョン
- Ollamaバージョン
- モデル一覧とcapabilities
- 有効設定
- 直近ターンの診断ログ
- エラースタック

個人パス、会話本文、認証情報は既定で含めない。

### 9.4 依存機能の段階的劣化

`pdftotext`がない場合でもサーバー全体を止めない現在の方針を、他機能にも適用する。

- vision非対応 → 画像機能だけ無効
- Gitなし → Git診断だけ無効
- Windows相互運用なし → Linux範囲だけ利用

## 10. 品質指標

改善効果を主観だけで判断せず、次を継続測定する。

- タスク完了率
- 空応答率
- 1タスクあたりの平均ツール呼び出し数
- 同一ツール反復による停止率
- 圧縮後の重要情報保持率
- ビルド・テスト未実施のまま終了した割合
- HTTP再試行成功率
- モデル別の平均所要時間
- ユーザー停止率
- 再送によって成功した割合

履歴から匿名のローカル統計を生成し、外部送信は行わない。

## 12. 到達点

LocalCoderの目標は、単に多くのツールを持つことではない。

```text
モデルが失敗する
    → 失敗を記録できる
    → 原因を再現できる
    → サーバー側で補正できる
    → 回帰テストで再発を防げる
```

可逆操作による安全性、機械的状態管理、診断可能性、実障害の回帰テストを組み合わせることで、LocalCoderは弱いローカルモデルを実用的な作業エージェントとして成立させる基盤になる。

## 13. 外部ツールプロバイダ (MCP) 対応

開発者が別途設計している「PicoCalc開発支援エキスパートMCP」（Google Drive上の設計文書、
本リポジトリ外）は、LocalCoderやCodex CLIなど複数のコーディングエージェントから共通利用
できる独立MCPサーバーとして、PicoCalc関連の資料・過去プロジェクト・ソースコードの検索、
診断、比較、レビューを提供する構想である。LocalCoder自身に固有ロジックを埋め込まず、
外部の専門知識基盤として接続する設計になっている。

### 13.1 外部文書が明示する前提条件

同文書は「LocalCoderへの導入タイミング」として、次をMCP接続より先に固めるべき基盤として
明示的に列挙している（本書の対応節と現状を併記する）。

| 外部文書の項目 | 本書の対応節 | 現状 |
|---|---|---|

### 13.2 ToolProviderインターフェース（新規概念）

現状の`exec_tool`は組み込みツール（`run_command`/`read_file`/`write_file`等）を`if name ==`
の連鎖で直接ディスパッチしている。MCPサーバーが提供するツールも同じ`tool_calls`ループへ
差し込むには、組み込みツールとMCPツールを同じ形で扱える薄い抽象化が要る。

```python
class ToolProvider(Protocol):
    def list_tools(self) -> list[dict]: ...       # Ollamaへ渡すtools定義の一部を返す
    def call_tool(self, name: str, args: dict, ctx: ToolContext) -> str: ...
```

組み込みツール群を`BuiltinToolProvider`として同じインターフェースに揃えた上で、MCP接続時に
`McpToolProvider`（`tools/list`・`tools/call`をラップ）を追加する。`exec_tool`は
`name → provider`のマップを引くだけにし、個々のツール実装の分岐ロジックには触れない。
これは§8.1（`server.py`の段階的分割）で`tools.py`を切り出す作業と自然に合流する。

### 13.3 役割分担（外部文書の記述を踏襲）

- **PicoCalc Expert MCP**: 資料検索・診断・比較・レビューのみ。読み取り・分析専用。
- **LocalCoder（クライアント側）**: 実際のファイル読み書き・コード変更・ビルド・テスト・
  実機確認の指示。MCPから得た調査結果を元に、既存ツール（`write_file`/`run_command`等）で
  実作業を行う。

この役割分担は、LocalCoderの既存原則「重要な状態はLLMの記憶ではなく機械的に保持する」
「基本機能はローカルで完結させる」と矛盾しない——PicoCalc Expert MCPも「LLMの記憶ではなく
索引化された資料と機械的検索」を基盤にする設計になっている（外部文書§17設計原則）。
