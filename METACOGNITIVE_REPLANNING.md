# 自動メタ認知・再計画パス設計

LocalCoder がローカルLLMの作業停滞や目的逸脱を検知し、通常作業へ自動的に割り込んで、現状の再構成・分析・計画更新を行わせるための設計計画。

## 1. 目的

LocalCoder は現在、ツール実行、履歴圧縮、作業状態の再構成、固定指示の保持、反復失敗検知、空応答回復、診断情報の保存を担っている。

これらはモデルの外部に置かれた記憶・監視・自己状態認識の補助層として機能している。次の段階として、観測した状態から「現在の方針を続けるべきか」を判定し、必要な時だけモデルに再計画を行わせる閉ループを追加する。

```text
観測
  ↓
停滞・逸脱・異常の判定
  ↓
現在状態の再構成
  ↓
計画の更新
  ↓
通常作業の再開
```

この機能をコード上では **replan pass**、UI上では「作業方針の自動見直し」と呼ぶ。

## 2. 基本方針

再計画は通常の会話ターンとは分離する。

- ユーザー発言として履歴へ追加しない
- 再計画中はツールを無効にする
- 再計画結果をユーザーへの最終回答にしない
- 結果は専用の内部状態として保存する
- 次の通常モデル呼び出しへ作業状態として差し込む
- 発火理由、計画、完了条件を診断ログへ記録する
- 必要な時だけ発火させ、定期的な無条件割り込みにはしない

再計画は「考え直してください」という曖昧な一文ではなく、目的、完了事項、未完了事項、障害、棄却仮説、次の最小検証、完了条件を構造化して出力させる。

## 3. 実行フロー

```text
通常のエージェントループ
    ↓
ツール結果と診断状態を更新
    ↓
should_replan(state)
    ├─ false → 通常ループを継続
    └─ true
         ↓
       build_replan_context(state)
         ↓
       ツールなしで run_replan_pass()
         ↓
       JSON結果を検証・正規化
         ↓
       current_plan と診断履歴へ保存
         ↓
       SSEでUIへ通知
         ↓
       次の通常ループで計画を再提示
```

再計画パス自体が失敗した場合は、通常作業を壊さず、エラーを診断ログへ残して従来のループへ戻る。JSONが壊れている場合は1回だけ修正要求を行い、それでも失敗した場合は再計画結果を採用しない。

## 4. 再計画プロンプト

再計画専用の system prompt を通常の `SYSTEM_PROMPT` とは別に持つ。

```text
これは作業継続のための自動再計画ターンです。
このターンではツールを使用せず、与えられた現在状態だけを分析してください。

以下を明確に整理してください。

1. ユーザーの最終目的
2. 現在までに完了したこと
3. まだ完了していないこと
4. 現在の障害または不確実な点
5. 直前の方針が進展しているか
6. 繰り返しているだけの操作がないか
7. 棄却すべき仮説
8. 次に試すべき最小の検証
9. その後の実行計画
10. 完了と判断するための条件

事実と推測を分離してください。
既に失敗した手順を理由なく繰り返さないでください。
変更済みファイルとユーザーの固定指示を維持してください。
出力は指定されたJSON形式だけにしてください。
```

期待する出力形式:

```json
{
  "goal": "最終目的",
  "completed": [
    "完了事項"
  ],
  "remaining": [
    "未完了事項"
  ],
  "blockers": [
    {
      "description": "現在の障害",
      "evidence": "その判断根拠"
    }
  ],
  "discarded_hypotheses": [
    "棄却する仮説"
  ],
  "next_step": {
    "action": "次に実行する最小の行動",
    "reason": "この行動を選ぶ理由",
    "expected_result": "期待する結果",
    "failure_means": "失敗時に分かること"
  },
  "plan": [
    "手順1",
    "手順2"
  ],
  "completion_criteria": [
    "完了条件"
  ]
}
```

## 5. 再計画へ渡すコンテキスト

履歴全体を再送せず、再計画に必要な状態だけを組み立てる。

- 現在のユーザー目的
- ユーザーの固定指示
- 現在の圧縮要約
- 前回の再計画結果
- 変更ファイル一覧
- 直近のツール呼び出しと結果
- 直近のエラー署名
- 開始時からのツール呼び出し数
- 最後の変更からのツール呼び出し数
- 圧縮回数
- 空応答・HTTP再試行回数
- 現在の推定トークン量
- 経過時間
- ビルド・テストの直近結果

```python
replan_messages = [
    {"role": "system", "content": REPLAN_SYSTEM_PROMPT},
    {"role": "user", "content": build_replan_context(state)},
]
```

`tools`は渡さない。モデルが再計画ターン中に調査や編集を始めることを防ぐ。

## 6. 発火条件

単一条件で即発火させるのではなく、複数の兆候をスコア化する。

### 6.1 ツール呼び出し数

```python
if tool_calls_since_replan >= 12:
    score += 2
```

長い探索で当初の目的より直近のエラーへ注意が偏っている可能性を示す。

### 6.2 変更または結論のない調査継続

```python
if tools_since_last_progress >= 8:
    score += 2
```

進捗はファイル変更だけで判定しない。次も進捗として扱う。

- 新しい根本原因の確定
- ビルドまたはテスト結果の改善
- 失敗仮説の棄却
- 必要ファイルや依存関係の特定
- 完了条件の一部達成

### 6.3 同一ツール失敗

同じツール名・引数の失敗が2回続いた時点で再計画候補にする。3回目まで同じなら従来通りstuckとして停止する。

```text
1回目: 通常継続
2回目: 再計画
3回目: 停止
```

### 6.4 同種エラーの反復

コマンドが異なっても同じ根本エラーが続く場合を検出する。

例:

- `undefined reference`
- `module not found`
- `permission denied`
- `file not found`
- 同一テストの同一失敗

```python
def error_signature(result: str) -> str:
    """パス、時刻、行番号、可変数値を除いて主要なエラーを正規化する。"""
```

同じ署名が3回出たら3点を加える。

### 6.5 履歴圧縮後

圧縮直後に必ず発火させるのではなく、次を満たす場合に1点を加える。

- そのターンで初回の圧縮
- 圧縮後にさらに4回以上ツールを使用
- 未完了状態

### 6.6 空応答回復後

空応答は文脈過多だけでなく、モデルが次の行動を選べなくなった兆候でもある。

```text
空応答
  ↓
強制圧縮
  ↓
再計画パス
  ↓
通常作業再開
```

空応答回復後は3点を加える。

### 6.7 長時間経過

経過時間だけで割り込ませず、進展不足と組み合わせる。

```python
if elapsed_seconds >= 600 and tools_since_last_progress >= 5:
    score += 1
```

### 6.8 同じファイルの再読

内容が変わっていない同一ファイルを3回以上読み直している場合は2点を加える。変更後の再読や、異なる範囲の明示的な再読は除外する。

### 6.9 大きな節目

停滞時だけでなく、次の成功後にも完了条件を再確認する。

- 初回ビルド成功
- 初回テスト成功
- 根本原因の確定
- 大規模変更の終了
- 主要依存関係の解決

節目発火は失敗回復ではなく、「部分成功を全体完了と誤認しない」ために使う。

## 7. 初期スコア設計

```python
REPLAN_SCORE_THRESHOLD = 4
REPLAN_AFTER_TOOL_CALLS = 12
REPLAN_NO_PROGRESS_TOOLS = 8
REPLAN_MIN_INTERVAL_TOOLS = 6
REPLAN_MAX_PER_TURN = 3


def replan_score(state) -> int:
    score = 0

    if state.tool_calls_since_replan >= REPLAN_AFTER_TOOL_CALLS:
        score += 2

    if state.tools_since_last_progress >= REPLAN_NO_PROGRESS_TOOLS:
        score += 2

    if state.same_tool_failure_count >= 2:
        score += 3

    if state.same_error_signature_count >= 3:
        score += 3

    if state.compacted_since_replan and state.tools_after_compaction >= 4:
        score += 1

    if state.empty_response_recovered:
        score += 3

    if state.elapsed_seconds >= 600 and state.tools_since_last_progress >= 5:
        score += 1

    if state.unchanged_file_reread_count >= 3:
        score += 2

    return score
```

閾値や重みは固定的な正解ではない。ターン診断情報へ発火前後の値を保存し、実セッションから調整する。

## 8. 連続発火の防止

再計画そのものが新しいループにならないよう制限する。

- 前回再計画から最低6ツール呼び出し空ける
- 1ターン最大3回
- 新しいツール結果が得られるまで再発火しない
- 前回と同じ発火理由だけでは再発火しない
- 再計画後も同じエラー署名が続いた場合は再計画回数を増やさず停止へ近づける
- 同一またはほぼ同一の計画が繰り返された場合はstuckと判定する

```text
再計画
  ↓
新計画で実行
  ↓
同じ障害が継続
  ↓
2回目の再計画
  ↓
依然として同じ
  ↓
障害と実施済み手順をユーザーへ提示して停止
```

## 9. 状態保存

再計画結果は通常の`user`メッセージとして保存しない。ユーザーが発言していない指示を履歴へ混入させないためである。

内部形式:

```json
{
  "role": "localcoder_meta",
  "meta_type": "replan",
  "created_at": 1783960000.0,
  "trigger": {
    "score": 5,
    "reasons": [
      "12 tool calls since last replan",
      "same error signature repeated 3 times"
    ]
  },
  "content": {
    "goal": "...",
    "completed": [],
    "remaining": [],
    "blockers": [],
    "next_step": {},
    "plan": [],
    "completion_criteria": []
  }
}
```

Ollamaへ送る時だけ、作業状態ダッシュボードへ次の形式で変換する。

```text
(自動再計画 - 現在の実行方針)

目的:
...

完了事項:
...

現在の障害:
...

次に行うこと:
1. ...
2. ...

完了条件:
...
```

履歴圧縮では最新の再計画結果を要約対象へ入れず、現在の作業計画として独立保持する。古い再計画結果は診断履歴として残すが、毎回のモデル入力には最新1件だけを使う。

## 10. 進捗の定義

再計画の精度は「進捗」をどう定義するかに依存する。ファイル変更だけを進捗とすると、調査・解析タスクで誤発火する。

初期実装では次を進捗イベントとする。

- `write_file` / `edit_file`の成功
- 直前よりビルドエラー数が減少
- 直前より失敗テスト数が減少
- 初めて成功したコマンド
- 新しいファイル・シンボル・依存関係を特定
- エラー署名が変化
- 再計画の`next_step`が完了
- 完了条件の達成数が増加

これらを`progress_events`として機械記録する。LLMが「進んだ」と自己申告しただけでは進捗扱いにしない。

## 11. UI

再計画が実行されたらSSEで通知する。

```json
{
  "type": "replan",
  "reason": "同じリンクエラーが3回続き、直近8ツールで進展がないため",
  "goal": "...",
  "next_step": "...",
  "completion_criteria": ["..."]
}
```

画面上では折りたたみカードとして表示する。

```text
🧭 作業方針を自動的に見直しました
理由: 同じリンクエラーが3回続きました
次: 実際にエクスポートされているシンボル名を確認します
[詳細]
```

詳細には次を表示する。

- 発火理由とスコア
- 現在の目的
- 完了事項
- 障害
- 棄却した仮説
- 次の最小検証
- 実行計画
- 完了条件

ターン終了時の作業サマリーには次を追加する。

- 再計画回数
- 最終再計画理由
- 再計画後に進捗が発生したか

## 12. 診断項目

既存のturn診断情報へ次を追加する。

```json
{
  "replan_count": 1,
  "replan_scores": [5],
  "replan_reasons": [["same_error", "no_progress"]],
  "tools_before_first_replan": 12,
  "progress_after_replan": true,
  "replan_model": "gpt-oss:20b",
  "replan_parse_retries": 0
}
```

品質指標として次を測定する。

- 再計画発火率
- 再計画後に進捗した割合
- 再計画後に完了した割合
- 再計画後も同じ障害で停止した割合
- 誤発火率
- 1タスクあたりの再計画回数
- 再計画による追加トークンと所要時間
- モデル別の再計画有効率

統計はローカルだけに保存し、外部送信しない。

## 13. モデル選択

初期実装では通常作業と同じモデルを使う。これによりモデル切替による文脈差や追加設定を避ける。

将来は次を設定可能にする。

```text
replan_model_policy:
  same_model
  fixed_model
  fallback_model
```

- `same_model`: 通常作業と同じモデル
- `fixed_model`: 再計画専用に指定したモデル
- `fallback_model`: 通常モデルが空応答やJSON失敗を起こした時だけ代替モデル

再計画専用モデルを使う場合も、ローカルOllama内で完結させる。

## 14. 失敗時の扱い

再計画パスは補助機構であり、本体の作業を停止させてはならない。

- Ollama HTTP失敗: 1回再試行し、失敗なら従来ループへ戻る
- 空応答: 1回だけJSON出力を促し、失敗なら採用しない
- JSON構文エラー: 生テキストを保存し、修正要求を1回行う
- 必須フィールド欠落: 空配列・空文字で正規化する
- 過大出力: 上限で切り詰め、計画項目数を制限する
- ツール呼び出しを返した: 実行せず、再計画失敗として記録する

再計画失敗そのものを再計画発火条件にはしない。

## 15. テスト計画

Ollamaを必要としない純粋関数から実装する。

追加候補:

```text
tests/
├── test_replan_scoring.py
├── test_replan_context.py
├── test_replan_parsing.py
├── test_replan_state.py
└── test_replan_integration.py
```

最低限のテスト:

- スコア閾値の境界
- 単一の弱い兆候では発火しない
- 複数兆候の組み合わせで発火する
- 空応答回復後の発火
- 圧縮直後だけでは発火しない
- 最小ツール間隔による連続発火防止
- 1ターン最大回数
- 同じエラー署名の正規化
- 同じファイル再読の検出
- 進捗イベントでカウンタがリセットされる
- 正常JSONの読み込み
- コードフェンス付きJSONの正規化
- 欠落フィールドの補完
- 壊れたJSONの再試行
- `localcoder_meta`がユーザー発言として保存されない
- 最新計画だけがモデル入力へ差し込まれる
- 再計画失敗時も通常ループが継続する
- 再計画後に同一障害が続く場合の停止判定

実障害fixtureとして、長時間探索、同一エラー反復、空応答回復、圧縮後の目的逸脱が起きたセッションを追加する。

## 16. 段階的実装

### 第1段階: 手動チェックポイント

自動判定を入れる前に、内部関数とUIを検証する。

1. `run_replan_pass()`
2. `build_replan_context()`
3. JSONパース・正規化
4. `current_plan`保存
5. SSEカード表示
6. 開発用の手動発火ボタンまたは内部フラグ

### 第2段階: 明確な異常から自動発火

1. 同一ツール失敗2回
2. 空応答回復後
3. 同種エラー3回
4. 最大回数・最小間隔
5. 診断情報保存

### 第3段階: 停滞検知

1. ツール呼び出し12回
2. 進捗イベント管理
3. 進展なし8ツール
4. 同一ファイル再読
5. 圧縮後の再確認

### 第4段階: 完了条件との統合

1. `goal`専用フィールド
2. `completion_criteria`
3. 完了条件の達成状態
4. 部分成功後の節目再計画
5. 未検証のまま終了するモデルへの完了拒否

### 第5段階: 実測による調整

1. モデル別の発火率
2. 再計画後の成功率
3. 誤発火の分析
4. スコアと閾値の調整
5. モデル別プロンプト
6. 任意の再計画専用モデル

## 17. 最初に実装する最小仕様

最初の実用品は次に限定する。

発火条件:

1. ツール呼び出しが12回に到達
2. 同じツール失敗が2回連続
3. 空応答から回復
4. 履歴圧縮後にさらに4回ツール使用
5. 直近8ツールで進捗イベントがない

動作:

1. ツールなしで再計画を1回実行
2. JSON形式で結果を受け取る
3. 最新計画として内部保存
4. 作業状態ダッシュボードへ追加
5. SSEでUIへ表示
6. 通常ループを継続
7. 1ターン最大2回、最低6ツール間隔

この仕様で実セッションを収集し、スコア方式へ拡張する。

## 18. LocalCoder全体での位置づけ

LocalCoderは次の三層を持つ。

```text
実行層
  ファイル、コマンド、Web、画像、PDF

記憶・状態層
  会話履歴、圧縮、変更ファイル、固定指示、診断情報

メタ認知・監督層
  文脈監視、反復検知、空応答回復、進捗評価、再計画
```

再計画パスを追加すると、メタ認知層は単なる異常検知ではなく、観測結果から行動方針を更新する制御層になる。

```text
モデルが作業する
    ↓
LocalCoderが状態を観測する
    ↓
停滞や目的逸脱を検知する
    ↓
モデル自身に現状を再構成させる
    ↓
新しい計画を固定する
    ↓
モデルが新計画に沿って作業を再開する
```

この機能の目的は、モデルへ長い追加思考を常時要求することではない。モデルが目的を見失い始めた時だけ、LocalCoderが外部監督者として立ち止まらせ、プロジェクト完遂へ向けて方針を再構築させることである。
