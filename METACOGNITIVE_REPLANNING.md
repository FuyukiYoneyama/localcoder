# 自動メタ認知・方針再評価パス設計

> 文書の役割: 方針再評価パスの設計根拠、判定モデル、実装状況を一体で管理する。文書全体を未実装計画とはみなさず、実装済み・未実装の区別は本文の進捗記録に従う。

LocalCoder がローカルLLMの作業停滞、目的逸脱、反復、文脈劣化を検知し、通常作業へ自動的に割り込んで、現状の再構成と方針の再評価を行わせるための設計計画。

> **実装上の最重要原則**
>
> この機能の目的は、必ず計画を変更することではない。
> 現在の方針が引き続き妥当かを検査し、`CONTINUE`、`ADJUST`、`CHANGE`、`STOP`のいずれかを明示的に判定することである。
>
> `CONTINUE`は「何もしなかった」ことを意味しない。根拠を調べた結果、現在の方針へ期限付きの継続許可を与えた、情報価値のある判定である。

コード上では既存名称との連続性を保つため **replan pass** を使用してよいが、概念上およびUI上では **方針再評価パス** または「作業方針の自動見直し」と呼ぶ。

## 1. 目的

LocalCoder は現在、ツール実行、履歴圧縮、作業状態の再構成、固定指示の保持、反復失敗検知、空応答回復、診断情報の保存を担っている。

これらはモデルの外側に置かれた記憶・監視・自己状態認識の補助層として機能している。次の段階として、観測した状態から現在の方針を再評価し、必要な制御を選択する閉ループを追加する。

```text
観測
  ↓
停滞・逸脱・異常・節目の判定
  ↓
現在状態と前回予測の再構成
  ↓
方針の再評価
  ↓
CONTINUE / ADJUST / CHANGE / STOP
  ↓
判断理由・期待結果・再評価条件を保存
  ↓
通常作業の再開または停止
```

この層の役割は、現在の方針を一定期間ごとに再認証することである。方針変更はその結果の一つにすぎない。

## 2. 判定モデル

再評価結果は必ず次の4種類のいずれかとする。

### 2.1 `CONTINUE`

現在の方針は妥当であり、指定された期間または条件まで継続する。

使用例:

- 停滞に見えたが必要な探索である
- 類似した操作だが異なる仮説を順番に検証している
- エラーは続いているが原因候補の切り分けは進んでいる
- 現在の手順が完了条件への最小コスト経路である

`CONTINUE`には必ず次を含める。

- 継続を正当化する具体的な証拠
- 次の行動
- その行動から期待する結果
- 期待が外れた場合に何が分かるか
- 有効期限または再評価条件

理由や期限のない`CONTINUE`は無効とする。

### 2.2 `ADJUST`

基本方針は維持するが、順序、対象範囲、検証方法、ツール選択などを修正する。

使用例:

- 実装方針は維持するが先にテストを追加する
- 原因仮説は維持するが別の確認方法を使う
- 全体方針は維持し対象ファイルを限定する
- 同じビルドを繰り返さず最小再現を作る

### 2.3 `CHANGE`

前提または方針が支持されなくなったため、別の計画へ切り替える。

使用例:

- 前回の予測が実績によって否定された
- 同じ障害が方針変更後も解消しない
- 完了条件へつながらない探索を続けている
- より低コストで確実な経路が判明した

### 2.4 `STOP`

自律的な継続が不適切または不可能であるため停止する。

使用例:

- 必要情報がなく推測で進めるしかない
- 外部サービスや環境障害が解消されない
- 不可逆操作または外部送信に承認が必要
- 再評価を繰り返しても新しい計画が得られない
- ユーザーの指示間に解消不能な矛盾がある

## 3. 基本方針

方針再評価は通常の会話ターンとは分離する。

- ユーザー発言として履歴へ追加しない
- 再評価中はツールを無効にする
- 再評価結果をユーザーへの最終回答にしない
- 結果は専用の内部状態として保存する
- 次の通常モデル呼び出しへ作業状態として差し込む
- 発火理由、判定、根拠、期待結果、再評価条件を診断ログへ記録する
- 必要な時だけ発火させ、無条件の定期割り込みにはしない
- `CONTINUE`を含む全判定をUIで確認可能にする

「考え直してください」という曖昧な一文ではなく、目的、完了事項、未完了事項、証拠、反証、前回予測との照合、次の行動、完了条件を構造化して出力させる。

## 4. 実行フロー

```text
通常のエージェントループ
    ↓
ツール結果・進捗イベント・診断状態を更新
    ↓
should_review_strategy(state)
    ├─ false → 通常ループを継続
    └─ true
         ↓
       build_review_context(state)
         ↓
       ツールなしで run_review_pass()
         ↓
       JSON結果を検証・正規化
         ↓
       validate_review_decision(state, result)
         ├─ 不十分 → 1回だけ修正要求
         └─ 有効
              ↓
            current_strategy_review と診断履歴へ保存
              ↓
            SSEでUIへ通知
              ↓
            CONTINUE / ADJUST / CHANGE → 通常ループ再開
            STOP → 障害と実施済み手順を提示して終了
```

再評価パス自体が失敗した場合は通常作業を壊さず、エラーを診断ログへ残して従来ループへ戻る。JSONが壊れている場合は1回だけ修正要求を行い、それでも失敗した場合は採用しない。

## 5. 再評価プロンプト

再評価専用のsystem promptを通常の`SYSTEM_PROMPT`とは別に持つ。

```text
これは作業継続のための自動方針再評価ターンです。
このターンではツールを使用せず、与えられた現在状態だけを分析してください。

重要:
- 必ず方針を変更する必要はありません。
- 現在の方針が妥当ならCONTINUEと判定してください。
- CONTINUEも明示的な判断であり、具体的証拠と有効期限が必要です。
- 基本方針を保ちながら手順だけ変える場合はADJUSTです。
- 前提や方針を変える場合はCHANGEです。
- 自律継続が不適切ならSTOPです。

以下を整理してください。

1. ユーザーの最終目的
2. 現在までに完了したこと
3. まだ完了していないこと
4. 現在の障害または不確実な点
5. 直前の方針が実際に進展しているか
6. 前回の再評価で予測した結果が実現したか
7. 現在の方針を支持する証拠
8. 現在の方針に反する証拠
9. 繰り返しているだけの操作がないか
10. 棄却すべき仮説
11. 次に行う最小の行動
12. その行動の期待結果と、失敗時に分かること
13. 完了と判断する条件
14. 次回の再評価時期または条件

事実と推測を分離してください。
既に失敗した手順を理由なく繰り返さないでください。
変更済みファイルとユーザーの固定指示を維持してください。
出力は指定されたJSON形式だけにしてください。
```

`decision`の許容値は小文字の`continue`、`adjust`、`change`、`stop`だけとする。

`confidence`はモデル自身の主観値であり、採用判断の唯一の根拠にはしない。履歴比較、UI表示、モデル別統計の補助値として扱う。

## 7. `CONTINUE`の採用条件

`CONTINUE`は次をすべて満たす場合だけ採用する。

1. `evidence`が1件以上ある
2. 直近の機械的状態と明白に矛盾しない
3. `next_step.action`が具体的である
4. `expected_result`が検証可能である
5. `failure_means`が記載されている
6. `review_after`に有限の再評価条件がある
7. 前回予測が外れた場合、その事実を無視して同じ理由を繰り返していない

次の場合は修正要求を返す。

```text
CONTINUEと判定していますが、直近の操作では新しい証拠が確認できません。
継続を正当化する具体的な進展を示すか、ADJUST、CHANGE、STOPへ判定を修正してください。
```

`CONTINUE`は無期限の許可ではない。`review_after`の条件を満たした時点で継続許可は失効し、再評価する。

## 8. 前回判断との照合

再評価入力には前回結果だけでなく、その後の実績を機械的に要約して含める。

```text
前回の判定: CONTINUE

前回の理由:
- 依存関係の切り分けが進んでいる
- 各操作から新しい証拠が得られている

前回の期待:
- 4ツール以内に不足ライブラリを特定する

その後の実績:
- ツール呼び出し: 4回
- 新規証拠: なし
- 除外した仮説: 0件
- 同じエラー署名: 3回
```

再評価の中心質問は次とする。

> 前回この方針を継続または変更すると判断した根拠は、その後の実績によって支持されたか。

前回予測が外れた場合は、それ自体を`counterevidence`として扱う。

## 9. 再評価へ渡すコンテキスト

履歴全体を再送せず、必要な状態だけを組み立てる。

- 現在のユーザー目的
- ユーザーの固定指示
- 現在の圧縮要約
- 前回の再評価結果
- 前回の期待結果と再評価条件
- 前回再評価後に起きた進捗イベント
- 変更ファイル一覧
- 直近のツール呼び出しと結果
- 直近のエラー署名
- 開始時および前回再評価後のツール呼び出し数
- 最後の進捗からのツール呼び出し数
- 圧縮回数
- 空応答・HTTP再試行回数
- 現在の推定トークン量
- 経過時間
- ビルド・テストの直近結果
- 完了条件の達成状況

```python
review_messages = [
    {"role": "system", "content": REVIEW_SYSTEM_PROMPT},
    {"role": "user", "content": build_review_context(state)},
]
```

`tools`は渡さない。再評価ターン中に調査や編集を始めることを防ぐ。

## 10. 進捗イベント

進捗はファイル変更だけで判定しない。次を機械的または半構造的な進捗イベントとして記録する。

- 新しい根本原因の確定
- ビルドまたはテスト結果の改善
- 失敗仮説の棄却
- 必要ファイルや依存関係の特定
- 完了条件の一部達成
- 新しいエラー署名の取得
- 前回の期待結果との一致
- 変更ファイルの増加
- 検証成功

可能なものはツール結果から機械的に導出する。LLMが申告しただけの進捗は、証拠となるツール結果または状態変化と関連付ける。

## 11. 発火条件

単一条件で即発火させるのではなく、複数の兆候をスコア化する。ただし`review_after`の期限到達は独立した強制発火条件とする。

### 11.1 ツール呼び出し数

```python
if tool_calls_since_review >= 12:
    score += 2
```

### 11.2 進捗のない調査継続

```python
if tools_since_last_progress >= 8:
    score += 2
```

### 11.3 同一ツール失敗

```text
1回目: 通常継続
2回目: 方針再評価
3回目: 原則停止。ただし再評価で具体的なADJUST/CHANGEが得られた場合のみ1回継続可能
```

### 11.4 同種エラーの反復

パス、時刻、行番号、可変数値を除いた`error_signature`を生成し、同じ署名が3回出たら3点を加える。

### 11.5 履歴圧縮後

次を満たす場合に1点を加える。

- そのターンで初回の圧縮
- 圧縮後にさらに4回以上ツールを使用
- 未完了状態

### 11.6 空応答回復後

空応答回復後は3点を加える。

```text
空応答
  ↓
強制圧縮
  ↓
方針再評価
  ↓
通常作業再開
```

### 11.7 長時間経過

```python
if elapsed_seconds >= 600 and tools_since_last_progress >= 5:
    score += 1
```

### 11.8 同じファイルの再読

内容が変わっていない同一ファイルを3回以上読み直した場合は2点を加える。変更後の再読や異なる範囲の再読は除外する。

### 11.9 大きな節目

停滞時だけでなく、次の成功後にも完了条件を再確認する。

- 初回ビルド成功
- 初回テスト成功
- 根本原因の確定
- 大規模変更の終了
- 主要依存関係の解決

### 11.10 `review_after`期限

前回判定の次のいずれかを満たした場合は、スコアに関係なく再評価する。

- 指定ツール呼び出し数へ到達
- 指定時間へ到達
- `replan_if`条件が成立
- `expected_progress`が期限内に得られなかった

## 12. 初期スコア設計

```python
REVIEW_SCORE_THRESHOLD = 4
REVIEW_AFTER_TOOL_CALLS = 12
REVIEW_NO_PROGRESS_TOOLS = 8
REVIEW_MIN_INTERVAL_TOOLS = 6
REVIEW_MAX_PER_TURN = 3


def review_score(state) -> int:
    score = 0

    if state.tool_calls_since_review >= REVIEW_AFTER_TOOL_CALLS:
        score += 2
    if state.tools_since_last_progress >= REVIEW_NO_PROGRESS_TOOLS:
        score += 2
    if state.same_tool_failure_count >= 2:
        score += 3
    if state.same_error_signature_count >= 3:
        score += 3
    if state.compacted_since_review and state.tools_after_compaction >= 4:
        score += 1
    if state.empty_response_recovered:
        score += 3
    if state.elapsed_seconds >= 600 and state.tools_since_last_progress >= 5:
        score += 1
    if state.unchanged_file_reread_count >= 3:
        score += 2

    return score
```

閾値や重みは実セッションの診断情報から調整する。

## 13. 連続発火と惰性継続の防止

- 前回再評価から原則6ツール呼び出し空ける
- 1ターン最大3回
- 新しいツール結果が得られるまで再発火しない
- `review_after`期限到達は最小間隔より優先する
- 前回と同じ発火理由だけでは再発火しない
- 同じ根拠の`CONTINUE`を、進捗なしで連続採用しない
- 再評価後も同じエラー署名が続いた場合は停止へ近づける
- 同一またはほぼ同一の計画が繰り返された場合はstuckと判定する

```text
CONTINUE
  ↓
期限内に期待した進展あり
  → 新しい根拠で再びCONTINUE可能

CONTINUE
  ↓
期限内に期待した進展なし
  → 同じ理由のCONTINUEは禁止
  → ADJUST / CHANGE / STOPを要求
```

## 15. UI

方針変更時だけでなく、`CONTINUE`を含むすべての判定を表示する。

```text
🧭 作業方針を自動的に見直しました

判定: 継続
理由:
・原因候補の切り分けが進んでいる
・直近の操作から新しい証拠が得られている

次の行動:
残る依存候補を検証

次回評価:
あと4回のツール実行後
または新しい証拠が得られない場合
```

カードには最低限次を表示する。

- 発火理由
- 判定
- 判断理由
- 反証または懸念
- 次の行動
- 期待結果
- 再評価条件

`CONTINUE`を表示することで、ユーザーは「一度検討したうえで現在の方向を続けている」ことを確認できる。

## 16. 診断情報

各ターンへ次を追加する。

- `strategy_review_count`
- `strategy_review_decisions`
- `last_strategy_review_at`
- `last_strategy_review_trigger`
- `review_json_retry_count`
- `continued_after_review`
- `adjusted_after_review`
- `changed_after_review`
- `stopped_after_review`
- `review_expected_progress_met`
- `review_prediction_accuracy`

品質指標として次を測定する。

- 判定別の割合
- `CONTINUE`後に期待進展が得られた割合
- `CONTINUE`後に同じ障害で再発火した割合
- `ADJUST` / `CHANGE`後の成功率
- 再評価なしでstuckした割合
- 再評価後も同じ計画を繰り返した割合
- 再評価の平均追加時間とトークン量

## 17. 段階的実装

### 第1段階: 最小仕様 【実装済み 2026-07-15】

- ツールなしの再評価呼び出し ✅ (`run_review_pass`。`REVIEW_SYSTEM_PROMPT`+
  機械的に組み立てた`build_review_context`のみを渡し、toolsは渡さない)
- 4種類の判定 ✅ (`continue`/`adjust`/`change`/`stop`。stopはターンを

2. `CONTINUE`後に進展せず、次回`ADJUST`
3. `ADJUST`後も同じエラーが続き`CHANGE`
4. 解消不能な障害で`STOP`
5. 方針が正しい長い探索を、不必要に変更せず完遂

## 19. 到達点

LocalCoderのメタ認知層は、方針変更装置ではない。

```text
現在の方針
    ↓
メタ認知による審査
    ↓
期限付き継続許可 / 条件付き修正 / 方針変更 / 停止
    ↓
期待結果を記録
    ↓
実績と照合
    ↓
再審査
```

この構造により、LocalCoderはモデルの方針をむやみに揺さぶらず、正しい方針を維持し、間違った方針だけを適切な時点で修正する外部監督制御層になる。
