# 外部MCP連携のセキュリティ設計と今後の課題

LocalCoderの現在のMCP対応は、利用者が管理する信頼済みローカルMCPサーバーをstdio子プロセスとして利用する**内部拡張機構**に限定している。本書は、この安全な既定値を維持したまま、将来HTTPS等を介した外部MCPサーバーと連携する場合に必要な設計・実装・検証事項を整理する。

## 1. 基本判断

外部MCP対応には将来性があるが、現在の内部MCPと同じ信頼境界で扱ってはならない。

```text
内部MCP
  信頼済みローカルコード
  stdio子プロセス
  既定で利用可能

外部MCP
  別の信頼領域
  HTTPS等のネットワーク通信
  認証・権限制御・承認・監査を必須化
```

外部MCPを導入しても、内部MCP限定モードを既定値として残す。外部接続は明示的に有効化した場合だけ利用可能にする。

## 2. 実装原則

- 内部MCPと外部MCPを別のProviderとして実装する
- セキュリティ判断をLLMへ委ねず、LocalCoder側の機械的ポリシーで決定する
- 認証情報をモデル、会話履歴、ツール引数、診断ログへ渡さない
- 読み取りと書き込みを分離し、外部読み取り専用から段階導入する
- 未分類のツールは安全側に倒し、自動実行しない
- ツール定義の変更を検知し、権限を自動継承しない
- 外部MCPの障害でLocalCoder本体を停止させない
- 外部MCPから得た文章は命令ではなく、非信頼データとして扱う

## 3. Providerの分離

現在の`McpToolProvider`はstdioローカルMCP専用として維持する。外部接続は別クラスとする。

```python
class StdioMcpToolProvider:
    """信頼済みローカルMCP専用。"""

class RemoteMcpToolProvider:
    """認証・ポリシー・監査を必要とする外部MCP専用。"""
```

JSON-RPCやツールスキーマ変換など、安全境界に関係しない共通処理だけを下位の共通クラスまたは純粋関数へ分離する。

内部向けの緩い条件が外部接続へ誤って適用されないことを、クラス境界とテストで保証する。

## 4. 信頼レベル

MCPサーバーごとに信頼レベルを明示する。

```text
local_trusted
remote_readonly
remote_scoped_write
remote_untrusted
disabled
```

### local_trusted

利用者が管理するローカルコードをstdio子プロセスとして起動する現在の方式。既定のMCPモード。

### remote_readonly

外部データの検索・取得だけを許可する。ファイル変更、投稿、送信、デプロイ等は許可しない。

### remote_scoped_write

明示された限定スコープで外部書き込みを許可する。操作単位の承認と監査ログを必須とする。

### remote_untrusted

ツール一覧の確認までは可能だが、自動実行は不可。明示的な管理者設定なしに昇格させない。

## 5. ツールの副作用分類

サーバー単位の信頼だけでは不十分である。同じサーバーでも、検索と削除では危険性が異なるため、各ツールをLocalCoder側で分類する。

```text
read_local
write_local_reversible
write_local_irreversible
external_read
external_write_reversible
external_write_irreversible
credential_operation
financial_operation
unknown
```

初期状態で`unknown`のツールは実行禁止または毎回承認とする。MCPサーバーが自己申告する分類だけを信用せず、利用者の設定またはLocalCoder側のポリシーで上書きできるようにする。

設定例:

```json
{
  "toolPolicies": {
    "company_docs.search": {
      "effect": "external_read",
      "approval": "none"
    },
    "github.create_issue": {
      "effect": "external_write_reversible",
      "approval": "required"
    },
    "github.delete_repository": {
      "effect": "external_write_irreversible",
      "approval": "always"
    }
  }
}
```

## 6. ポリシーエンジン

外部MCP呼び出しの直前に、LocalCoder側で必ずポリシー評価を行う。

```python
decision = policy_engine.evaluate(
    server=server,
    tool=tool,
    arguments=args,
    workspace=workspace,
    pinned_instructions=pinned_instructions,
)
```

判定は次の4種類とする。

```text
allow
allow_redacted
ask_user
deny
```

### 最低限確認する項目

- サーバーの信頼レベル
- ツールの副作用分類
- ユーザーの固定指示
- 外部送信禁止設定
- 引数に含まれる機密情報
- 作業フォルダ外の情報
- 認証情報・環境変数・秘密鍵らしき値
- 操作の可逆性
- 同一操作の反復
- 送信データ量と呼び出し回数

LLMやメタ認知層は補助的なリスク評価を提供できるが、最終的な許可・拒否はポリシーエンジンが決定する。

## 7. 認証情報の隔離

アクセストークン、クライアントシークレット、リフレッシュトークン等を次へ入れない。

- system prompt
- ユーザーメッセージ
- ツール引数
- 会話履歴
- 作業状態ダッシュボード
- MCP結果
- 診断パッケージ

認証情報はLocalCoderの資格情報ストアで管理し、通信層だけが参照する。

```text
LLM
  ↓ ツール名と業務引数のみ
LocalCoder policy / auth layer
  ↓ 認証ヘッダーを付加
Remote MCP
```

OSの資格情報ストアを利用できない環境では、最低限、権限を制限した専用ファイルへ保存し、通常の設定エクスポートや診断出力から除外する。

## 8. OAuthを導入する場合の要件

外部MCPの認可には、少なくとも次を必要とする。

- Authorization Code Flow + PKCE
- `state`の生成と検証
- 明示的なredirect URI
- 短寿命アクセストークン
- サーバーごとのトークン分離
- 最小スコープ
- リフレッシュトークンの安全な保存
- 失効・ログアウト処理
- トークンのログ・履歴・例外文からの除外

初期実装では、動的クライアント登録より固定登録または利用者による明示登録を優先する。

## 9. データ流出対策

外部MCPへ送る引数を、送信前に検査する。

検出対象例:

- APIキー、アクセストークン、秘密鍵
- `.env`や資格情報ファイルの内容
- 会話全文
- 未指定のソースコード
- 個人パス、ユーザー名、メールアドレス
- ユーザーが外部送信禁止と指定した情報

必要に応じて、値の拒否、マスキング、要約、ハッシュ化を行う。

外部MCP結果に含まれる文章は、モデルへの追加指示として扱わない。ツール結果を明確な非信頼データ境界で囲み、そこに記載された「別のツールを実行せよ」「秘密情報を送信せよ」等の命令を自動実行しない。

## 10. 承認ポリシー

承認はサーバー単位ではなく操作単位で行う。

```text
external_read
  設定により自動許可可能

external_write_reversible
  初回またはセッション単位で確認

external_write_irreversible
  毎回確認

credential_operation / financial_operation
  既定で禁止

unknown
  禁止または毎回確認
```

承認画面には次を表示する。

- 接続先サーバー
- ツール名
- 操作の副作用分類
- 送信する情報の要約
- 変更対象
- 可逆性
- 認証スコープ

LLMが生成した説明だけでなく、LocalCoderが引数とポリシーから機械的に生成した情報を表示する。

## 11. ツール定義の変更検知

外部MCPサーバーは、同じ名前のツールの説明や入力スキーマを後から変更できる。次を正規化してハッシュ保存する。

- サーバー識別子
- ツール名
- 説明
- 入力スキーマ
- 副作用分類
- 要求スコープ

前回承認時から変更があれば、そのツールの自動許可を停止し、再確認を要求する。

## 12. 監査ログ

外部MCP呼び出しは、会話履歴とは別の監査ログへ記録する。

```json
{
  "server": "github",
  "tool": "create_issue",
  "effect": "external_write_reversible",
  "arguments_digest": "sha256:...",
  "redacted_fields": ["body"],
  "policy_decision": "user_approved",
  "started_at": "...",
  "ended_at": "...",
  "result": "success"
}
```

秘密情報や本文をそのまま保存する必要はない。少なくとも次を追跡可能にする。

- どのサーバーとツールを使ったか
- どの権限・スコープで実行したか
- どのポリシー判断で許可されたか
- ユーザー承認があったか
- 成功、失敗、タイムアウト
- 呼び出し時間とデータ量

## 13. リモート通信の要件

外部MCPを実装する場合は、stdio実装へURL分岐を追加するのではなく、専用通信層を持つ。

最低限の要件:

- HTTPS必須
- 平文HTTPを既定で拒否
- 証明書検証を無効化できない
- 接続先ホストの明示的な許可リスト
- リダイレクト先の再検査
- DNS再解決やlocalhostへのリダイレクトを考慮したSSRF対策
- 応答サイズ上限
- 接続・読み取り・全体タイムアウト
- 呼び出し回数制限
- エラー本文の切り詰め
- 認証ヘッダーを別ホストへ転送しない

## 14. 名前空間

複数MCPサーバーを扱う場合、ツール名衝突を単なる先着順だけで処理しない。

```text
builtin__read_file
picocalc__get_reference
github__create_issue
```

表示名と内部識別子を分けてもよい。監査ログ、ポリシー、承認では必ずサーバーを含む完全修飾名を使う。

## 15. メタ認知層との連携

外部書き込み前に、方針再評価層へ次を補助的に評価させることができる。

- 最終目的に必要な操作か
- ローカル操作だけで代替できないか
- 現在の証拠で実行してよいか
- 不可逆性に見合う確信があるか
- ユーザーの固定指示に反していないか

ただし、メタ認知LLMの`CONTINUE`や`CHANGE`判定をセキュリティ許可として使わない。メタ認知はポリシー判断への入力の一つに留める。

## 16. 段階的ロードマップ

### Phase 0: 現状維持

- ✅ stdio内部MCPのみ
- ✅ 利用者が管理するローカル子プロセス
- ✅ 設定がなければMCP無効
- ✅ 起動・呼び出し失敗時の段階的劣化

### Phase 1: ポリシー基盤

- ⬜ MCPサーバーの信頼レベル
- ⬜ ツール副作用分類
- ⬜ `allow / allow_redacted / ask_user / deny`判定
- ⬜ 外部送信禁止設定との統合
- ⬜ 完全修飾ツール名
- ⬜ MCP呼び出しの構造化メタデータ
- ⬜ 監査ログ

この段階ではまだ外部MCPへ接続しない。内部MCPに対してポリシー機構を試験し、既存動作を壊さないことを確認する。

### Phase 2: 外部読み取り専用MCP

- ⬜ `RemoteMcpToolProvider`
- ⬜ HTTPS通信
- ⬜ 固定サーバー登録
- ⬜ OAuth + PKCEまたは同等の限定認証
- ⬜ 認証情報の隔離
- ⬜ 外部検索・ドキュメント取得だけを許可
- ⬜ 外部コンテンツを非信頼データとして扱う
- ⬜ 応答・データ量・呼び出し回数の制限

### Phase 3: 限定的な外部書き込み

- ⬜ 操作単位の承認
- ⬜ 引数プレビュー
- ⬜ 機密情報検出とマスキング
- ⬜ 可逆・不可逆操作の区別
- ⬜ 最小OAuthスコープの検証
- ⬜ ツール定義変更時の再承認

### Phase 4: 条件付き自律実行

- ⬜ 事前定義した低リスク操作だけを自動許可
- ⬜ メタ認知層による目的整合性の補助評価
- ⬜ 異常な呼び出し連鎖の検出
- ⬜ 費用、回数、データ量の上限
- ⬜ 自動許可の期限と再認証

## 17. テスト要件

最低限、次を自動テストする。

- 内部MCPの既存動作が変わらない
- 外部MCPが既定で無効
- 平文HTTPを拒否する
- 証明書検証を無効化できない
- 未分類ツールが自動実行されない
- 外部書き込みが承認なしで実行されない
- 認証情報がモデル入力・履歴・ログへ入らない
- 秘密らしき引数が拒否またはマスクされる
- リダイレクト後もホスト制約を維持する
- タイムアウト後に認証情報や子処理が残らない
- ツールスキーマ変更で自動許可が解除される
- 外部結果中の命令文が追加指示として処理されない
- 監査ログが秘密本文を保存しない
- ポリシーエンジン障害時は安全側に倒れる

## 18. 実装着手条件

外部MCP接続の実装は、少なくとも次が揃ってから開始する。

1. 副作用分類のデータ構造
2. ポリシーエンジン
3. 認証情報の安全な保存方式
4. 監査ログ
5. 操作単位の承認UI
6. 外部送信禁止設定との統合
7. セキュリティ回帰テスト

接続機能を先に実装して、後からセキュリティを付け足す順序は採用しない。

## 19. 到達点

目標は、LocalCoderのローカル完結性を失うことではない。

```text
安全な既定値
  内部MCPのみ
       ↓
明示的な高制約モード
  外部読み取り
       ↓
操作単位の許可
  限定外部書き込み
```

内部MCPは低摩擦なローカル拡張として維持し、外部MCPは認証・副作用分類・ポリシー・承認・監査を備えた別の信頼領域として追加する。これにより、現在の安全性を保ちながら将来のサービス連携へ拡張できる。
