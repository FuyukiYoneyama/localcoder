# 変更履歴 (Changelog)

LocalCoder の主な変更を時系列で記録する。形式は
[Keep a Changelog](https://keepachangelog.com/ja/1.1.0/) に準拠。
各項目の末尾 `(コミット)` は Git のコミットハッシュ。詳しい設計判断・検証結果は
各コミットメッセージと `REBUILD.md` の「設計ノート」を参照。

分類: **追加**=新機能 / **変更**=既存挙動の変更 / **修正**=バグ修正 /
**セキュリティ**=脆弱性対応 / **ドキュメント**=文書のみ。

---

## 2026-07-14

### ドキュメント
- **`IMPROVEMENTS.md`の実装順序を、外部のPicoCalc Expert MCP構想・
  `METACOGNITIVE_REPLANNING.md`と整合するよう見直した。** 開発者がGoogle Drive上に
  作成したPicoCalc Expert MCP設計文書が、MCP接続の前提としてLocalCoder側の
  特定の機能(テスト基盤・現在ゴール・完了条件の機械的管理・構造化ツール結果・
  ツール実行の分離・ToolProvider相当の共通インターフェース)を明示していた。
  `METACOGNITIVE_REPLANNING.md`の方針再評価パスも同じ前提(完了条件)を使う設計
  になっており、両構想が共有する未着手項目(現在のゴール専用フィールド§3.1、
  完了条件の機械的管理§3.3)の優先度を上げ、新規§13として外部MCP対応の前提
  チェックリストと`ToolProvider`インターフェース案を追記した。各項目の実装
  済み/未着手も現状に合わせて更新した。コード変更なし、ロードマップ文書のみ。
  (`81a2ed3`)

### 追加
- **完了条件の機械的管理を追加(`IMPROVEMENTS.md` §3.3の実装、MCP対応前提の
  第1弾)。** モデルが実際にはビルド/テストを実行していないのに「完了しました」
  と申告する事故を防ぐため、`find_unverified_changes`が`write_file`/`edit_file`
  で変更したファイルのうち、その後`run_command`を一度も実行していないものを
  機械的に検出するようにした(検証の成否は問わない、試みたかどうかのみ判定)。
  作業状態ダッシュボードに「⚠ 未検証の変更…」行を追加し毎ターン明示するほか、
  ターン診断情報(`unverified_changes_count`)・作業サマリーカード
  (「⚠ 未検証: Nファイル」、ホバーでファイル一覧)にも反映した。実チャットで
  `write_file`のみ実行させ、診断情報とUI表示の両方に正しく反映されることを
  確認済み。(`9217012`)
- **現在のゴール専用フィールドを追加(`IMPROVEMENTS.md` §3.1の実装、MCP対応前提の
  第2弾)。** ゴールはユーザー発言から機械的に抽出できないため、既存の要約
  プロンプト(どのみち圧縮のたびに1回呼ばれる)に相乗りする形にした。出力の
  1行目を`GOAL: <現在の最終目標を1文で>`という固定形式で必ず出力させ、
  `_extract_goal_line`で抽出して要約本文から取り除き、マーカーの専用ブロックに
  保存する。新しい判定が得られなければ前回のゴールを引き継ぐ(ファイル一覧・
  固定指示と同じ方式)。`build_work_state`に「現在のゴール: …」行を追加(圧縮が
  一度も起きていない会話では非表示)。開発中に`GOAL_LINE_RE`の`\s*`が改行にも
  マッチしGOAL値が空の場合に次の行を巻き込むバグを発見し、`[ \t]*`に修正して
  テストで固定した。`gpt-oss:20b`・`qwen2.5-coder:7b`の両方で実際に
  `GOAL: ...`形式が正しく出力・抽出されることを確認済み。(`729e708`)

### 変更
- **ツール実行部分を`BuiltinToolProvider`クラスへ分離し、`ToolProvider`共通
  インターフェースを導入した(`IMPROVEMENTS.md` §8.1の一部＋§13.2の実装、
  MCP対応前提の第3弾)。** 組み込みツールの分岐ロジックを`exec_tool`から
  `BuiltinToolProvider.call_tool()`へ移し、`list_tools()`/`call_tool()`という
  共通形式(`ToolProvider` Protocol)に揃えた。ツール実行に必要な状態は
  `ToolContext`に1つにまとめ、ツール名からプロバイダを探す`_provider_for_tool`
  はデータ駆動(if/elifではなく`list_tools()`との突き合わせ)にした。将来
  `McpToolProvider`を`TOOL_PROVIDERS`へ追加するだけで組み込みツールと同列に
  扱えるようにする受け皿。`exec_tool`自体は公開シグネチャ・挙動を変えない薄い
  ラッパーとして残し、既存93テストは無修正で全パス。実チャットで
  `write_file`→`read_file`→`run_command`の3種を連続実行させ新しいディスパッチ
  経路の動作を確認済み。物理的な`server.py`の複数ファイル分割は今回のスコープ外
  (第5段階で別途行う)。(`abe2904`)
- **REBUILD.mdに埋め込まれた`server.py`/`index.html`の完全ソースコピーを
  再同期した。** 今日の一連の変更(コミット多数)に追従できておらず、埋め込み
  コピーには`find_unverified_changes`等の関数が存在しない状態だった。
  埋め込みコードを抽出して実ファイルとdiffし、0差分(完全一致)を確認。(`4fecb2e`)
- **構造化ツール結果を追加した(`IMPROVEMENTS.md` §4.1の実装、MCP対応前提の
  最終項目)。** 当初は既存の文字列規約の書き換えリスクを見込んで見送っていたが、
  §4.1の原文が「既存の文字列結果に**加えて**構造化データを保持する」という
  追加的な設計を求めているだけだと気づき、リスクを最小化した形で実装した。
  `run_command`/`ToolProvider`/`exec_tool`いずれも戻り値の型は変えず、新しい
  純粋関数`parse_command_result`が既存の文字列表現(`"exit_code=N..."`等)から
  構造化データを事後的に導出する。`build_work_state`の素朴な
  `startswith("exit_code=0")`チェック2箇所をこれに置き換え(挙動は完全互換)、
  `handle_chat`は`run_command`の結果メッセージに`meta`フィールド(構造化データ)
  を追記するようにした(`content`は変更せず追加のみ)。stdout/stderrの分離や
  `duration_ms`は、現在の実装から安全に導出できないため見送った。これで
  MCP対応前提の5項目(現在のゴール・完了条件の機械的管理・構造化ツール結果・
  ツール実行の分離・ToolProviderインターフェース)が全て完了し、第6段階
  (MCP接続)に着手できる状態になった。実チャットで成功/失敗コマンドを実行させ、
  `meta`フィールドが正しく記録されることを確認済み。

## 2026-07-13

### 追加
- **`tests/`に自動テスト一式を追加(`IMPROVEMENTS.md` §2.1〜2.2の実装)。** 標準
  ライブラリの`unittest`のみで完結し(pytest等の外部依存を増やさない)、Ollama
  接続も不要(`ollama_ask`は`FakeOllama`スタブで差し替え)。対象は履歴圧縮
  (重複除去・ヒステリシス・世代劣化防止・先回り圧縮・強制圧縮)、作業状態
  ダッシュボード、ツール名の正規化、連続失敗検出、履歴タイトルの永続化。
  `tests/fixtures/`には実際に問題が起きた4セッションのJSON(パスのみ匿名化した
  実データ)を配置し、各不具合の再発を検知する回帰テストとした。ただし
  `tests/fixtures/`自体は`.gitignore`対象とし、実データはこのマシンにのみ
  存在してGitHubへは一切コミットしない(会話内容の実データをリポジトリに含める
  べきではないという判断から)。fixtureが無い環境では該当テストのみ
  `unittest.SkipTest`で自動スキップされ、スイート全体は失敗しない。
  - この過程で`track_tool_repeat`のオフバイワンを発見・修正した。「同じ呼び出しが
    3回連続失敗したら打ち切る」という設計意図に反し、実装は4回目でようやく
    打ち切っていた(初回の失敗をカウント0として扱っていたため)。実害は
    「1回多く待ってから打ち切る」程度で軽微だが、テストを書いたことで発覚した。
  - ツール名の特殊トークン除去(`sanitize_tool_name`)と連続失敗判定
    (`track_tool_repeat`)を、`do_POST`内のインライン処理から純粋関数として
    切り出した(挙動は変えず、テスト可能にするための移動)。
  - 実行方法: `python3 -m unittest discover -s tests -t .`（詳細は`REBUILD.md`
    §4-1）。(`3b66a17`)
- **ターン診断情報の保存とセッションJSONの`schema_version`を追加
  (`IMPROVEMENTS.md` §2.3/§8.3の実装)。** 障害調査のたびに履歴JSONから手計算
  していたトークン見積もり・圧縮回数・ツール呼び出し数などを、`handle_chat`が
  各ターンで機械的に記録するようにした。`turn`辞書に
  `est_tokens_start`/`est_tokens_end`・`compact_count`・`http_retries`・
  `empty_retries`・`tool_call_count`・`tool_exec_seconds`・`iterations_used`
  を追加。あわせて履歴JSONに`schema_version`(現在2)を導入し、今後形式を
  変える際に安全に移行できるようにした。実チャットで動作確認済み。(`11f322e`)
- **`run_command`の出力が切り詰められても、完全な内容を診断用に保存する
  ようにした(`IMPROVEMENTS.md` §4.2の一部実装)。** 12KBを超える出力は
  モデル向けに先頭6KB+末尾6KBへ切り詰めていたが、その切り詰め後の文字列が
  そのまま会話履歴にも保存されるため、中間にあるビルドエラー等が永久に
  失われていた。切り詰めが発生した場合のみ、完全な出力を
  `history/tool_output/<sid>/<call_id>.txt`へ保存するようにした
  (`history/`配下なので`.gitignore`で自動的に除外され、GitHubには上がらない)。
  構造化ツール結果(§4.1、`{ok, exit_code, stdout, stderr, ...}`形式)への
  全面移行は、既存の`ERROR:`/`OK:`/`exit_code=0`という文字列規約が広範囲に
  定着・テスト済みのため今回は見送った。実チャットで動作確認済み。(`10129d1`)
- **ターン終了時にチャットへ作業サマリーカードを表示するようにした
  (`IMPROVEMENTS.md` §7.1の実装)。** 直前で保存できるようにした診断情報が、
  今までは履歴JSONを直接読まないと見えなかった。新しいSSEイベント
  `{"type":"summary"}`で「終了理由・所要時間・ツール呼び出し回数・変更
  ファイル数・圧縮回数」を送り、フロント側でカード表示するようにした
  (変更ファイルの一覧はカードのホバーで確認可能)。過去の履歴を開いた時にも
  直近ターンのサマリーを表示する。診断情報を持たない旧履歴では0件表示に
  フォールバックする。実ブラウザで、ライブ実行時・履歴再表示時の両方を
  確認済み。(`172cc3e`)
- **起動時セルフチェックを追加(`IMPROVEMENTS.md` §9.2の実装)。** Ollama接続
  可否・推奨モデル(`gpt-oss`/`ornith`系統)の有無・`pdftotext`の有無・
  `ALLOWED_ROOTS`の各パスが実在するか・履歴ディレクトリへの書き込み可否を
  起動時に1回確認し、問題があればブラウザ上部にバナー表示するようにした
  (クリックで詳細展開)。「直したはずが再起動を忘れて古いプロセスのままだった」
  「Ollamaが落ちているのに気づかず作業を始めていた」といった事故の早期発見が
  狙い。全項目失敗してもサーバー起動は止めない(既存の「依存機能の段階的劣化」
  方針に合わせた)。新設`GET /api/selfcheck`はトークン不要。実ブラウザで、
  正常時にバナーが出ないこと・`LOCALCODER_OLLAMA`を疎通不可にした状態で
  バナー表示とクリック展開が正しく動くことを確認済み。(`5175329`)
- **ユーザーの「覚えておいて」等の継続指示を、圧縮を跨いで一字一句保持する
  ようにした(`IMPROVEMENTS.md` §3.2の実装)。** これまでは要約プロンプトが
  「一字一句そのまま残すこと」と指示するだけで、LLMが従うかどうかに委ねて
  いた。変更ファイル一覧と同じ「機械抽出→圧縮マーカーの専用ブロックで要約
  対象外にする」方式を適用し、`extract_pinned_instructions`が「覚えて」
  「忘れないで」を含むユーザー発言を機械的に集めて`PINNED_SECTION_HEADER`
  ブロックに引き継ぐようにした。`_parse_marker`/`build_marker`は
  `(summary, files, pinned)`の3値を扱うよう拡張。あわせて作業状態ダッシュ
  ボードにも「ユーザーの固定指示(厳守):」行を追加し、圧縮されていない
  直近ウィンドウでも毎ターン思い出させるようにした。「現在のゴール専用
  フィールド」(§3.1)は、機械抽出できずLLM要約ステップが別途必要になるため
  今回のスコープからは外した。(`65d55fd`)

## 2026-07-12

### 変更
- **履歴自動圧縮: 要約の世代劣化を防ぐインクリメンタル更新に変更。** 圧縮が
  複数回走ると、前回の要約自体を毎回生ログとして丸ごと再要約してしまい、
  繰り返すほど内容(特にファイルパスや作業の方向性)が薄まる問題があった。
  既存の圧縮マーカーを検出した場合は再要約せず「これまでの要約」として
  引き継ぎ、新規分の会話だけを要約してLLMにマージさせるようにした。変更
  ファイル一覧はLLMを介さず機械抽出し、要約本文とは別ブロックとして常に
  正確な値のまま引き継ぐようにした(要約に失敗しても消えない)。未着手の
  追加アイディア(ユーザー指示の永続ピン留め・3層の段階的減衰・ゴール専用
  フィールド)は `REBUILD.md` の設計ノートに記録した。(`653a593`)

### 修正
- **モデルが漏らす特殊トークンでツール呼び出しが永久に失敗し続ける不具合を修正。**
  一部のモデル(実例: `laguna-xs-2.1`)がツール呼び出しの`name`欄に
  `<|tool_call_argument_begin|>`のような内部特殊トークンを混入させて返すことがあり、
  完全一致ディスパッチが常に失敗し続けて最大ループ回数(80)を浪費し、次のターンで
  再開しても同じ失敗が再発する現象が実際に発生していた。`<|...|>`パターンを
  ツール名から除去してから実行するようにし、正規化後の名前を履歴にも書き戻す
  ようにした。
  - あわせて、作業状態ダッシュボードの変更ファイル一覧が`write_file`/`edit_file`の
    呼び出しがあっただけで成功とみなしていたため、実際には書き込みに失敗している
    のにモデルが「ファイルは作成済み」と誤認するハルシネーションを誘発していた。
    ツール結果が`ERROR`で始まる場合は変更ファイル一覧に含めないようにした。
  - 同じツール呼び出し(名前+引数)が同じエラーで3回連続失敗したら、最大ループ回数
    まで待たずにその場で打ち切り、原因を明示するメッセージを出すようにした。
  - `write_file`等で必須引数が欠けている場合のエラーメッセージを、生の`KeyError`
    ではなく「どの引数が足りないか」を明示する内容に変更し、モデルが自己修正
    しやすくした。(`d9ceace`)
- **履歴自動圧縮が天井に張り付いて頻発し、作業が進まなくなる不具合を修正。**
  実測で、73KBの履歴のうち37KBが同一内容のツール結果の重複(弱いモデルが同じ
  ファイルを読み直し続けた結果)だったセッションで、会話が予算(24576トークン)の
  天井に張り付いて数イテレーションごとに圧縮が再発し、切り詰めのたびに履歴前方が
  書き換わってollamaのプロンプトキャッシュが毎回無効化される(→全プロンプト再処理で
  数分停止)ため実質作業が止まる現象が起きていた。対策:
  - `dedupe_tool_results`(LLM不使用)を追加。同一内容のツール結果は最新の1件だけ
    残し、古い方は短い参照文に置換する(該当セッションで24168→14605トークン)。
  - 圧縮にヒステリシスを導入。発動判定は予算超過のままだが、発動したら
    予算の`COMPACT_TARGET_RATIO`(0.6)まで一気に下げ、天井張り付き→即再発の
    ループを断つようにした。
  - 要約更新を「新規分の要約→既存要約とマージ」の2回のLLM呼び出しから
    `update_summary`による1回に統合し、圧縮1回あたりの停止時間を半減させた。
  - 作業状態ダッシュボードに、同じツール呼び出し(名前+引数)が3回連続した場合の
    警告を追加(成功していても進展のない繰り返しをモデルに自覚させる)。(`e6c5948`)
- **予算天井付近でモデルが空応答を繰り返し、何も進まなくなる不具合を修正。**
  実測で、書き込み系ツールを一度も呼ばずドキュメント読解だけで会話が予算
  (24576トークン)の99%まで伸び、その状態でモデルが本文もツール呼び出しも
  無い空応答を2回連続(自動リトライ1回を使い切っても)返して停止するセッションが
  発生していた。弱いモデルは巨大な文脈下で生成用の余白を思考に使い切り本文が
  一切残らないことがある一方、従来の空応答リトライは「続けてください」を
  1文足すだけで文脈量をほぼ減らさず、同じ壁に再度当たるだけだった。対策:
  - `compact_history`の発動判定を「予算超過時」から「予算×
    `PROACTIVE_COMPACT_RATIO`(0.9)を超えた時点」に前倒しし、正式に超過する前に
    先回りで縮めるようにした。実際に空応答が起きたセッションを再生し、この
    しきい値なら空応答発生前に発動していたことを確認済み。
  - 空応答の自動リトライ直前に`compact_history(..., force=True)`を呼び、
    しきい値未満でも強制的に圧縮してから再試行するようにした。(`2cae068`)

### 追加
- **画面ヘッダーに実行中バージョン(gitコミットハッシュ)を表示。** 「修正して
  pushしたのに直っていない」という報告を調べたところ、原因はLocalCoderの
  再起動を忘れて古いプロセスのまま動かし続けていたことだった。起動時に
  `git rev-parse --short=7 HEAD`で自身のコミットを取得し、`git log`の最新と
  見比べるだけで最新版が動いているか一目で分かるようにした(未コミットの
  変更があれば`+dirty`を付記、git実行に失敗したら"unknown")。

### 修正
- **履歴一覧のタイトルが、圧縮が起きたセッションはどれも同じ定型文になり
  区別できなくなる不具合を修正。** タイトルは「最初のuserメッセージ」から
  毎回作り直していたため、圧縮でその位置が「【自動要約】ここまでの会話が
  長くなったため…」という定型マーカーに置き換わると、以後ずっとそのタイトルに
  なっていた。タイトルは初回保存時に一度だけ確定して以後は固定するようにし、
  マーカーだった場合は要約本文の冒頭を使うようにした。既存の壊れたタイトルを
  持つ履歴ファイルも同じロジックで一度だけ移行した。(`c871075`)

## 2026-07-11

### 追加
- **画像・PDFファイルの扱いに対応。** 専用の添付UIは設けず、既存のファイル
  ツール群を拡張する形で実装した(ワークスペース配下のファイルに自由にアクセス
  できる既存設計と一貫させるため)。
  - `read_file` が `.pdf` を自動判別し `pdftotext`(poppler-utils)でテキスト
    抽出するようにした。スキャン画像PDF等でテキストが取れない場合はその旨を返す。
  - 新規ツール `view_image`(png/jpg/gif/webp/bmp)を追加。現在選択中のモデルが
    vision対応か `/api/show` の `capabilities` で判定し、対応していれば
    Ollamaへの次回呼び出しで実際に画像を見せる(合成のuserメッセージとして
    `images` フィールドに載せる)。非対応モデルではエラーを返し、エージェントが
    ユーザーにモデル切り替えを促せるようにした。
  - `/api/models` がモデルごとの vision 対応可否も返すようになり、画面上部の
    モデル選択の横に「👁 画像対応 / 🚫 画像非対応」バッジを表示するようにした。
  - 画像はチャット欄にサムネイル表示。履歴の圧縮/要約では画像の中身を対象に
    含めず「画像N枚が添付されていた」という短いマーカーに留め、トークン見積もりにも
    画像1枚あたりの概算コストを加算するようにした。(`20cf464`)
- **Windows のファイル操作・コマンド実行に対応。** 作業フォルダとファイルツール
  (read_file/write_file/edit_file/list_dir) を WSLホーム(`$HOME`)に加え Windows
  ドライブ(`/mnt/c` 等)配下でも許可し、`C:\...` のファイルを直接編集できるように
  した。Windows コマンドは run_command から `powershell.exe` で実行する
  (WSL相互運用。元々実行自体は可能だった)。許可範囲は環境変数
  `LOCALCODER_ALLOWED_ROOTS` で変更可 (`$HOME` だけに戻せる)。(`e8eefb4`)

## 2026-07-05

### 追加
- **作業状態ダッシュボード。** 変更したファイル一覧・直近コマンドの成否・同一
  コマンドの3回連続失敗警告を、履歴から機械的に(LLM不使用で)組み立て、その回の
  Ollama呼び出しにのみ差し込む(保存履歴は汚さない)。小型モデルの「変更を忘れる」
  「同じ失敗を繰り返す」対策。(`e58a774`)
- **フォルダ選択ダイアログ。** 作業フォルダ欄の「📁 参照」ボタンでモーダルの
  ディレクトリブラウザを開く (`GET /api/browse`、要トークン)。(`9c71bfe`)
- **完了ノーティス。** リクエスト完了で入力待ちに戻ると、ベル音(Web Audio)＋
  デスクトップ通知(Notification API)で知らせる。停止ボタン押下時は通知しない。(`5ce5c9a`)
- **作業時刻の記録。** プロンプト受信時刻と完了/停止/エラー時刻を
  `history/<sid>.json` の `turns` 配列に記録 (status: completed/stopped/
  max_iter/error/disconnected)。(`fce11f8`)
- **空応答からの自動回復。** モデルが本文なし・ツール呼び出しなしで黙って
  止まった場合、1回だけ自動で続行を促し、それでも空ならユーザーに通知。(`6269903`)
- **Ollama呼び出し失敗時の自動再試行。** HTTP 500 等で失敗したら2秒待って1回だけ
  自動再試行。一過性のGPU/VRAM負荷を手動再送せず乗り越える。(`fdd4a25`)

### 変更
- **システムプロンプトの規律強化。** 実運用で繰り返し観測された3つの失敗
  (与えられた正確な値の捏造 / 存在しないライブラリ・関数名の使用 / ビルド検証を
  せずユーザーに丸投げ) への対抗ルールを明記。(`0a7100b`)
- 1リクエストあたりの最大ツールループ回数 `MAX_ITER` を 40 → 80 に引き上げ
  (実履歴で37〜39回に達する例があったため)。(`5ce5c9a`)
- フォルダ一覧の並び順を大文字小文字を無視した名前順に変更。(`411ce7d`)

## 2026-07-04

### 追加
- **edit_file ツール。** 完全一致の find/replace による部分編集。既存ファイルの
  修正は全文書き換え(write_file)より優先され、トークン節約と書き換え漏れ防止に
  なる。(`c5a676d`)
- **履歴の自動圧縮。** 会話がコンテキスト長(32K)に近づくと、古いツール結果の
  切り詰め → 古い会話のLLM要約への置換を自動で行う。長い会話でもシステム
  プロンプトや直近文脈が押し出されて壊れない。(`3803c1b`)
