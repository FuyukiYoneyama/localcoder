# 変更履歴 (Changelog)

LocalCoder の主な変更を時系列で記録する。形式は
[Keep a Changelog](https://keepachangelog.com/ja/1.1.0/) に準拠。
各項目の末尾 `(コミット)` は Git のコミットハッシュ。詳しい設計判断・検証結果は
各コミットメッセージと `REBUILD.md` の「設計ノート」を参照。

分類: **追加**=新機能 / **変更**=既存挙動の変更 / **修正**=バグ修正 /
**セキュリティ**=脆弱性対応 / **ドキュメント**=文書のみ。

## 2026-07-16

### 追加
- **方針再評価に「前回予測との照合」(第3段階)とエラー署名正規化を追加した
  (`METACOGNITIVE_REPLANNING.md` 第3段階全項目+第4段階の一部)。** 実障害:
  SDKパスがワークスペース外にあるため同種のパスエラーが1ターンに17回発生した
  が、毎回パスが違うため完全一致の連続失敗検知に乗らず、合間の書き込み成功で
  進捗なしカウンタもリセットされ続け、再評価は3回ともCONTINUEのままmax_iter
  (80イテレーション)まで走った。しかも2回目のCONTINUEは「前回もcontinueと
  判断された」ことを根拠に挙げる自己強化に陥っていた。対策: (1)パス・数値を
  正規化したエラー署名で同種エラーを数え3回で+3点(`same_error`)。(2)進捗
  イベントを種類付きで記録し、「前回判定後の実績」(ツール数・進捗件数・
  エラー反復)と中心質問を再評価コンテキストへ機械的に組み込む。(3)出力JSONに
  `previous_review.prediction_met`(前回期待の達成判定)を要求しUI・ダッシュ
  ボードに表示。(4)**前回がCONTINUEで進捗イベント0件のまま再びCONTINUEなら
  機械的に不採用**とし修正要求でadjust/change/stopへの変更を促す。あわせて
  1ターン発火上限を採用数→試行数ベースに修正(不採用が続くと上限が効かず
  再評価LLMが毎ツール後に走り続ける潜在バグ)し、不採用直後のクールダウンを
  追加。単体テスト16件追加(計261件)。実Ollama(gpt-oss:20b)で、前ターンの
  CONTINUE(期待:main.c作成)を仕込んだ停滞タスクが`prediction_met: false`の
  正しい照合結果を含むCHANGE判定に格上げされることを確認済み。(`212ca19`)

## 2026-07-15

### 修正
- **未検証の変更を残したままツール呼び出し無しで終了する問題を修正した。**
  ターン跨ぎ引き継ぎ修正の直後の実セッションで、モデルが8ファイルを実際に
  作成した後、main.cppもビルド検証も行わないまま「残りのファイルを作成して
  ビルドする」という平文だけを返してターンが`completed`扱いで終了する事例が
  起きた。方針再評価はツール呼び出しループの内側でしか発火しないため、
  ツール呼び出し自体が無いこの終了パターンは検知できなかった。既存の空応答
  自動回復(`EMPTY_RETRY_LIMIT`)と同じ枠組みで、「ツール呼び出し無し・本文あり
  だが`find_unverified_changes`が1件以上ある」場合に1回だけ自動で続行を促す
  `UNFINISHED_RETRY_LIMIT`を追加(既存の未検証変更検知をそのまま流用、新しい
  判定ロジックは増やしていない)。空応答と違い文脈枯渇が原因ではないため
  強制圧縮はしない。診断情報に`unfinished_retries`を追加。実Ollama
  (gpt-oss:20b)で、未検証の書き込み済み合成履歴+「ツール無しで一文報告して」
  という指示により再現: 通知とnudgeメッセージの履歴挿入を確認、モデルは
  その後run_command等で検証作業を自発的に再開し`completed`で正常終了した
  (1回のnudgeで無限ループにならないことも確認)。(`62e65ea`)
- **方針再評価の判定がターンをまたぐと消える問題を修正した。** リリース直後の
  実セッションで、ターン0のSTOP判定が「CMakeLists.txtとmain.cを1回で作成する」
  という具体的な計画を残したのに、次ターンでユーザーが「実装を開始して」と

  ターン開始時に履歴中の最後の判定をシードして最初のOllama呼び出しから
  ダッシュボードに提示する(前ターン由来は「新しい指示を優先」ラベル付き)。
  `review_after`期限の時計はターン開始から数え直し(古いセッション再開での
  即時発火を防止)、同一理由の再発火抑止・1ターン発火予算は引き継がない。
  あわせて`plan`(複数手順)と`completion_criteria`(完了条件)をダッシュボードと

  1ツール目に`review_after_due`で発火することを実Ollamaで確認済み。

### 追加
- **自動方針再評価パスを追加した(`METACOGNITIVE_REPLANNING.md` 第1〜2段階の
  実装)。** 実障害セッションで「74ツール呼び出し・72イテレーションを消費して
  タスク本体のmain.cppを一度も作らず、『まず確認してから作成します』という
  宣言と確認だけを30回以上繰り返す」というエラーゼロの停滞(既存の同一失敗
  検知では原理的に捕まらない)が起き、ユーザーが手動停止する事例があった。
  機械的カウンタ(`ReviewState`)からスコア(12ツール+2/進捗なし8ツール+2/
  同一失敗2回+3/空応答回復+3/圧縮後4ツール+1/経過10分+1/内容不変の再読3回+2、
  閾値4点)で発火を判定し、ツールなしの専用LLM呼び出しが現在状態の機械的
  要約だけを見て`continue`/`adjust`/`change`/`stop`を構造化JSONで判定する。
  CONTINUEには根拠・具体的な次の行動・検証可能な期待結果・有限の再評価期限を
  機械的に必須化(不備は1回だけ修正要求、それでも駄目なら不採用で通常続行)。
  進捗はLLMの申告ではなくツール結果から機械的に判定(ファイル変更系の成功と
  失敗コマンドの回復のみ)。採用結果はrole=`localcoder_meta`として履歴に永続化
  しつつOllama入力からは除外し、次呼び出しの作業状態ダッシュボードへ「現在の
  実行方針」として再提示。STOP判定はターンを停止する(`review_stop`)。UIは
  CONTINUEを含む全判定を🧭カードで表示(履歴再表示にも対応)。
  `LOCALCODER_STRATEGY_REVIEW=off`で無効化可。単体テスト39件追加(計234件)。
  実Ollama(gpt-oss:20b)で発火→有効なCONTINUE→履歴保存→再提示の全経路を確認済み。
  (`b7550b2`)
- **可逆操作レイヤー第3段階(一部): 外部送信の分類とポリシーを追加した
  (`REVERSIBLE_OPERATIONS.md` §7-8/第3段階の§3・§5)。** 「危険なのは取り消せない
  ネットへの書き込み」という設計原則の中核。`classify_external_send(cmd)`が
  run_commandのコマンド文字列から外部送信(git push / curl・wgetのPOST系・
  アップロード / scp・sftp / rsync・sshのリモート / npm・twine・gh release・
  docker push等の公開 / aws s3・gsutilアップロード / メール送信)を検出する
  (取得=GETは安全側として対象外。難読化・変数展開は捕捉外の安全網)。環境変数
  `LOCALCODER_EXTERNAL_SEND_POLICY`で挙動を選べる: `allow_recorded`(既定・
  従来通り実行するが送信内容を台帳へ必ず記録)/ `deny`(実行前に拒否しモデルに
  ユーザー依頼を促す)。外部送信は`manifest.json`の`external_sends`(コマンド全文・
  検出理由・実行有無)に記録され、サマリーカードに「📤 外部送信: N件」を表示、
  起動時セルフチェックにポリシー項目を追加。`ask`(実行前のUI同期確認)はSSE往復
  承認が必要なため未実装、run_command前後のファイル差分検出(§7-B)は観測用で
  可逆化に直結しないため後回し。単体テスト9件追加(計195件)。deny/allow_recorded
  両方をサーバー起動込みで実チャット確認済み。(`381b583`)
- **可逆操作レイヤー第2段階: 削除・移動・コピーを可逆化した
  (`REVERSIBLE_OPERATIONS.md` §5-6/第2段階の実装)。** 専用ツール
  `delete_file`/`delete_directory`/`move_file`/`copy_file`を追加。削除は即時
  消去せずトランザクションの`trash/`へ退避(ディレクトリは配下の全ファイルを
  相対パスごと保存し、サブツリーを丸ごと復元可能)。移動は移動元・移動先・
  上書きされた既存内容を記録。ロールバックは操作を逆順に適用し(delete→復元、
  move→逆移動、上書き既存も復元)、再適用(redo)は再度削除/移動する。システム
  プロンプトで「`rm`/`mv`より専用ツールを優先。dedicatedツールなら1ターンを
  丸ごとundoできるが生の`rm`は戻せない」と明示した(run_command自体は従来通り
  サンドボックス無しで実行可能——第3段階でdiff検出を追加予定)。台帳領域
  (`.localcoder/`)の削除・移動先指定は拒否、ワークスペースルート自体の削除も拒否。
  サマリーカードの変更ファイル数はdelete/move/copyのdstも数える。単体テスト
  14件追加(計186件)。実チャットのAPI通しで delete+move → ロールバックで
  両方復元(undeleted=1/moved_back=1)を確認済み。(`fe4869a`)
- **可逆操作レイヤー第1段階: ファイル編集をターン単位で元に戻せるようにした
  (`REVERSIBLE_OPERATIONS.md` §13/第1段階の実装)。** 「セキュリティ上問題に
  なるのは手戻りできない書き込みだけ」という設計原則に基づき、1回の
  `/api/chat`リクエストを1トランザクションとして扱う`Transaction`台帳を導入
  した。`write_file`/`edit_file`は書き込み直前に変更前状態(内容・権限・
  更新時刻・SHA-256)をワークスペース配下`.localcoder/transactions/<id>/before/`
  へ保存し(同一ファイルは最初の1回だけ§3)、書き込み自体も一時ファイル+
  `os.replace`の原子的書き込みに変更した(§4.2)。manifest.jsonは操作のたびに
  更新されるため、ターン途中のクラッシュでも台帳は有効なまま。新設
  `POST /api/transaction/rollback`(要トークン)が操作を逆順に取り消し、新規
  作成ファイルの削除・空になった新規親ディレクトリの除去まで行う。
  ロールバック自体も可逆で、直前の状態を`after/`へ退避してから戻すため
  `POST /api/transaction/reapply`でredoできる(undo/redoの往復可)。UIの
  ターン終了サマリーカードに「⎌ このターンの変更を元に戻す」ボタンを追加し、
  履歴を開き直した後でも同じ操作ができる(`turns`に`txn_id`を記録)。
  読み取りだけのターンではディスクに何も作られず、`.localcoder/`は
  自己無視の`.gitignore`を持つためユーザーのgitリポジトリを汚さない。
  台帳領域自体への`write_file`/`edit_file`は拒否(可逆性の保証を守るため)。
  停止・エラー時の自動ロールバックはしない(§10、途中までの変更が有益な場合が
  あるためユーザーが選ぶ)。単体テスト26件を追加(計172件)。実チャット
  (glm-4.7:cloud)で既存ファイル変更+新規作成→APIロールバックで完全復元
  (restored=1/removed=1)→再適用、の往復を確認済み。(`460be55`)
- **MCPクライアント機能を追加し、PicoCalc Expert MCPを接続した(`IMPROVEMENTS.md`
  §13 / 第6段階の実装)。** 新しい`McpToolProvider`が外部MCPサーバー(stdio /
  JSON-RPC 2.0)を子プロセスとして起動し、そのツールを組み込みツールと同列に
  モデルへ提供する。第3.5段階で用意した`ToolProvider`インターフェースの2つ目の

  `mcp_servers.json.example`をコミット)で、ファイルが無ければ従来通り。
  トランスポートはstdioのみでマシン内完結のため外部送信ポリシー
  (`REVERSIBLE_OPERATIONS.md` §8)の対象外(HTTP/SSEは導入時に再検討)。
  起動失敗・呼び出し失敗は段階的劣化(空のツール一覧・`ERROR:`文字列)で吸収し、
  名前衝突は組み込み優先。セルフチェックに「MCP設定」項目、診断パッケージに
  `mcp_servers`(名前・生存・ツール数のみ)を追加。フェイクMCPサーバーを実際に
  子プロセス起動する単体テスト16件を追加(計146件)。実チャットで
  `get_reference(topic='psram')`がMCP経由で呼ばれ正本文書が返ることを確認済み。
  (`377083f`)
